## 写在前面

最初买 VPS 时，我的目标很简单：拥有一台可以自由部署服务、全天在线的远程服务器。

真正开始使用后，我才发现“成功运行”只是第一步。网络会波动，证书会过期，服务可能意外退出，错误配置也可能把自己挡在服务器外。于是，这台机器逐渐从一次安装实验，变成了一套包含部署、安全、监控、备份和故障恢复的个人运维实践。

这篇文章记录的不是一份万能脚本，而是我如何把一台刚买来的 VPS，一步步变成可以长期维护的系统。出于安全考虑，文中不会出现真实 IP、端口、密钥或服务凭据。

## VPS 是什么

VPS（Virtual Private Server，虚拟专用服务器）可以理解为一台放在数据中心、全天开机并拥有公网网络的虚拟电脑。它没有显示器，日常通过 SSH 远程管理。

| 对比 | 个人电脑 | VPS |
|---|---|---|
| 所在位置 | 家中或宿舍 | 数据中心 |
| 运行时间 | 随使用开关机 | 通常全天运行 |
| 网络 | 多数位于局域网 | 通常拥有公网地址 |
| 管理方式 | 图形界面 | SSH 和控制台 |
| 典型用途 | 日常使用 | 网站、自建服务、实验环境 |

我选择的是一台位于洛杉矶的入门型 Linux VPS。配置不高，但对于轻量服务、网络实验和 Linux 学习已经足够。系统使用 Ubuntu LTS，以减少版本频繁变化带来的维护成本。

## 第一阶段：先让服务器可控

拿到服务器后，我没有立即安装各种应用，而是先确认系统和资源状态：

```bash
cat /etc/os-release
free -h
df -h
uptime
```

随后更新软件包，并设置正确的时区：

```bash
apt update && apt upgrade -y
timedatectl set-timezone Asia/Shanghai
```

这一阶段最重要的不是安装了什么，而是建立一个基本习惯：每次修改前先了解现状，每次修改后立即验证结果。

## 第二阶段：从密码登录迁移到 SSH 密钥

服务器最初允许 root 密码登录，使用方便，但也意味着公网扫描器可以持续尝试猜测密码。后来我生成了独立的 Ed25519 密钥，在确认密钥登录成功后关闭密码认证。

我还做了这些限制：

- 将 SSH 服务迁移到非默认端口，减少无意义的扫描日志。
- 限制单次连接的认证尝试次数和登录等待时间。
- 使用 fail2ban 自动封禁短时间内反复失败的来源。
- 保留服务商控制台作为 SSH 完全失效时的应急入口。

这里有一个很重要的操作顺序：**先放行新端口并验证新连接，再关闭旧入口。** 我曾因为顺序错误短暂失去 SSH 连接，最后只能通过服务商控制台修复。

## 第三阶段：最小化公网暴露

我使用 UFW 管理防火墙，原则是默认拒绝所有入站连接，只开放确实需要从公网访问的服务：

```bash
ufw default deny incoming
ufw default allow outgoing
ufw enable
```

最初，为了方便操作，我把管理面板直接开放在公网。即使修改了默认端口、路径和密码，它仍然是一个不必要的攻击面。

现在的方案是让面板只监听本机，通过 SSH 隧道访问：

```text
我的电脑 ── SSH 加密隧道 ── VPS 本地管理面板
```

这样，只有先通过 SSH 密钥认证的设备才能接触管理界面。订阅端口、废弃节点和不再使用的大范围 UDP 规则也都从公网关闭。

## 第四阶段：网络方案的几次迭代

这部分经历了最多的试错。

最早使用的方案可以连接，但实际速度很低。我先在服务器端测试下载能力，确认机房带宽本身没有问题，再把排查范围缩小到传输协议、拥塞控制和本地网络路径。

之后我尝试过 Hysteria2，并调整 BBR、UDP 缓冲区和 QUIC 窗口。速度一度得到非常明显的改善，这也让我理解了一个重要事实：服务器性能、线路质量和传输协议是三个不同的问题，不能只凭延迟判断体验。

但速度不是唯一指标。经历网络不稳定和客户端兼容问题后，我重新评估了维护成本。目前采用的结构是：

- **主用线路**：VLESS + WebSocket + TLS，直接连接服务器。
- **备用线路**：经 Cloudflare 中转，在直连异常时作为恢复通道。
- **Hysteria2**：保留配置但暂停运行，不再开放对应防火墙规则。

最终选择不一定是跑分最高的方案，却是我当前环境中兼容性、稳定性与维护难度更平衡的方案。

## 第五阶段：让服务能够自我恢复

手动重启服务只能解决一次故障。为了让服务器在无人值守时也能保持基本可用，我给关键服务补上了自动恢复机制。

服务进程停止时，systemd 会尝试重新启动；健康检查则定时检查进程、监听端口和基础资源状态。这里我刻意划定了自动化边界：

- 确认进程停止，可以自动重启。
- 端口异常、内存过高或 CPU 波动，只发送告警，不贸然修改配置。
- 连续两次检测异常后才通知，减少偶发波动产生的误报。
- 服务恢复后发送恢复消息，让故障形成闭环。

自动化不是越激进越好。一个判断错误的“自动修复”，有时会比原始故障造成更大影响。

## 第六阶段：监控、日志与通知

现在服务器会定时生成状态摘要，包含：

- 关键服务与端口状态
- 系统负载、内存和磁盘占用
- 启动时间与当前连接数
- 本次开机以来的网络流量
- SSH 失败次数与 fail2ban 封禁情况
- TLS 证书剩余有效期
- 待更新的软件包数量
- 最近一次配置备份状态

日报和异常通知会发送到我的消息机器人。需要注意的是，这种通知由 VPS 自己发出：服务挂掉但服务器仍能联网时可以报警；如果整台机器宕机或彻底断网，它也无法开口。因此，外部可用性监控仍然是后续需要补上的一环。

日志本身也需要维护。我为自动任务和系统日志设置了大小限制、轮转和保留周期，避免监控系统反过来把硬盘写满。

## 第七阶段：把备份当成恢复流程

“有备份文件”不等于“能够恢复”。现在服务器会定时备份关键配置，并只保留一定数量的历史版本，避免无限占用空间。

备份完成后还会检查：

1. 最新备份是否在合理时间内生成。
2. 文件大小是否异常。
3. 压缩包能否正常读取。
4. 数据库、证书和关键配置是否确实包含在内。

除了服务器本地备份，我还保存了一份经过系统账户加密的 Windows 异地备份，并实际验证过它能够解密。真正可靠的备份至少需要回答三个问题：它是否足够新、是否完整，以及能否恢复。

## 我踩过的坑

### 先改 SSH，再配防火墙

结果是新端口没有放行，旧连接断开。正确顺序应该是放行、验证，再关闭旧端口。

### 直接修改面板数据库

数据库内容变化不代表运行中的服务一定重新加载配置。不同版本还可能存在缓存和表结构差异。涉及节点配置时，应优先使用面板提供的管理界面，并在修改后检查真实监听状态。

### 证书文件权限错误

服务通常不会以 root 身份运行。如果证书放在普通服务账户无权读取的位置，即使路径正确也会启动失败。解决问题时要同时检查文件路径、所有者和权限。

### 把 Cloudflare 模式当成一个开关

CDN 回源涉及端口、TLS、SNI、Host 和证书匹配，任何一项不一致都可能表现为连接失败。排查时应该逐层确认，而不是不断随机切换选项。

### 只关注服务，不关注恢复

服务部署成功的那一刻最轻松，但真正花时间的是几天后发生的故障。后来我开始在部署时同时考虑检查命令、备份位置、告警方式和恢复入口。

## 现在的架构

```text
个人设备
  ├─ 主用加密线路 ───────────→ VPS 服务
  └─ Cloudflare 备用线路 ────→ VPS 服务

管理设备
  └─ SSH 密钥 + 本地隧道 ───→ 管理面板

VPS
  ├─ UFW + fail2ban
  ├─ 自动安全更新
  ├─ 健康检查与故障通知
  ├─ 日志轮转
  └─ 本地备份 + 加密异地备份
```

这套架构并不复杂，但每一层都对应一个真实遇到过的问题。

## 总结

这次 VPS 实践让我学到的，远不止怎样安装某个服务。

我开始理解最小权限、故障边界和可恢复性；学会先收集证据，再修改配置；也意识到“更快”不等于“更适合长期使用”。一台可靠的服务器，不是从未发生故障，而是在故障发生时能够被发现、被定位，并且有办法恢复。

这台 VPS 仍然是一个持续迭代的项目。接下来，我会补充独立的外部监控，并继续把新的问题和解决过程整理到这里。
