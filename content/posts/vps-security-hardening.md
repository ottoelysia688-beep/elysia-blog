## 写在前面

买这台 VPS 之前，我对服务器的理解大概只有一句话：**它是一台放在别人机房里的电脑。**

SSH 是什么、端口为什么要开放、证书放在哪里，我都不清楚。最开始我只想搭一个自己用的服务，没想到后来先后遇到了速度只有 20kbps、SSH 断连、证书没有权限、IP 网络异常、备用线路配置错误等问题。

现在回头看，真正让我学会东西的不是某条“一键安装命令”，而是每次出问题后怎样判断：到底是服务器坏了、程序没启动、防火墙挡住了，还是网络线路本身出了问题。

所以这篇文章会从最基础的地方开始。你不需要预先懂 Linux，我会尽量解释每一步是在做什么，以及我为什么这样做。

> 为了安全，文中的域名、端口和密码都使用示例值。不要直接照抄示例凭据，也不要把自己的节点链接、服务器 IP 或私钥发到网上。

## 一、VPS 到底是什么

VPS 的全称是 Virtual Private Server，中文一般叫“虚拟专用服务器”。

可以把它理解为一台：

- 放在数据中心里；
- 一天 24 小时开机；
- 通常有公网 IP；
- 没有显示器，需要远程操作的电脑。

服务商把一台性能很强的物理服务器分成很多台虚拟服务器，我们购买其中的一小份。它和普通电脑的区别大概是这样：

| 项目 | 家里的电脑 | VPS |
|---|---|---|
| 放在哪里 | 家里或宿舍 | 服务商的数据中心 |
| 怎么操作 | 鼠标、键盘和显示器 | 通过 SSH 远程登录 |
| 是否长期在线 | 关机后就离线 | 一般全天运行 |
| 网络地址 | 通常在家庭局域网内 | 通常拥有公网 IP |
| 常见用途 | 日常学习和娱乐 | 网站、自建服务、程序实验 |

我购买的是一台位于洛杉矶的入门 VPS，配置是 1 核 CPU、1GB 内存和十几 GB 硬盘。听起来很弱，但运行几个轻量服务其实足够了。

### 为什么选择 Ubuntu

购买 VPS 时需要选择操作系统。我选的是 Ubuntu LTS，因为网上资料多，遇到问题比较容易搜到答案。

LTS 是长期支持版本的意思。对于刚接触服务器的人来说，稳定通常比追求最新更重要。

## 二、购买时应该看什么

刚开始我也只会看价格，后来才知道一台 VPS 是否好用，至少要看下面几项：

| 项目 | 我的理解 |
|---|---|
| CPU 和内存 | 决定能同时运行多少程序 |
| 硬盘 | 系统和服务文件放在哪里 |
| 月流量 | 一个月允许传输多少数据 |
| 端口带宽 | 理论上最快能传多快 |
| 机房位置 | 会影响延迟和线路质量 |
| 是否提供控制台 | SSH 失效时能不能进去抢救 |

我最后选择了便宜的年付套餐。对于第一次尝试的人，我不建议一上来购买很贵的配置。先用低成本机器弄明白自己真正需要什么，比盲目追求“高性能线路”更有用。

付款后，服务商会通过邮件提供服务器 IP、初始用户名和密码。请把它们存放在安全的位置，不要截图公开，也不要继续长期使用邮件里的初始密码。

## 三、第一次登录服务器

### SSH 是什么

SSH 是远程管理服务器的一种加密连接方式。

当我们在自己的电脑上连接 VPS 后，看到的黑色命令行窗口实际上是在操作远方服务器，而不是本机。

Windows 现在自带 SSH。在 PowerShell 中可以这样连接：

```powershell
ssh root@你的服务器IP
```

第一次连接时会询问是否信任这台服务器，确认 IP 无误后输入 `yes`，然后输入服务商给的初始密码。

输入密码时屏幕上不会出现星号，这是正常的。成功后，命令行提示符通常会变成类似：

```text
root@server:~#
```

这表示已经进入 VPS。

### 登录后先别急着安装东西

我现在拿到新服务器后，会先运行下面几条命令：

```bash
# 查看系统版本
cat /etc/os-release

# 查看内存
free -h

# 查看硬盘
df -h

# 查看服务器运行时间和负载
uptime
```

然后更新系统：

```bash
apt update && apt upgrade -y
```

这些命令的作用不是“优化”，而是先确认买到的配置是否正确、系统有没有明显问题，并安装已经发布的安全更新。

## 四、我的第一个问题：为什么速度只有 20kbps

最初把服务安装好时，我以为事情已经结束了。客户端可以连接，网页也偶尔能打开，但速度只有大约 20kbps，看视频几乎不可能。

我的第一反应是：这台便宜 VPS 的网络太差。

但如果直接换服务器，我就不会知道真正的问题在哪里。所以先在 VPS 上测试它自己下载文件的速度：

```bash
curl -o /dev/null -w 'Speed: %{speed_download} bytes/s\n' \
  https://speed.cloudflare.com/__down?bytes=100000000
```

测试结果显示，服务器本身的下载速度并不慢。这说明：

```text
服务器访问互联网：正常
我的电脑通过服务访问互联网：很慢
```

问题更可能出现在我和 VPS 之间的传输，而不是 VPS 的硬件性能。

### 检查 BBR

BBR 是 Linux 的一种 TCP 拥塞控制算法。简单理解，它负责判断网络中应该以多快的速度发送数据。

查看当前算法：

```bash
sysctl net.ipv4.tcp_congestion_control
```

如果系统支持 BBR，可以配置：

```bash
cat >> /etc/sysctl.conf <<'EOF'
net.core.default_qdisc=fq
net.ipv4.tcp_congestion_control=bbr
EOF

sysctl -p
```

再次确认：

```bash
sysctl net.ipv4.tcp_congestion_control
```

看到输出为 `bbr` 才表示生效。

开启 BBR 后速度有改善，但仍然没有达到服务器本身应有的水平。这让我意识到，单纯复制一条“网络优化命令”并不能解决所有问题。

## 五、从 TCP 到 Hysteria2：第一次大幅提速

最开始使用的是基于 TCP 的方案。后来我尝试了 Hysteria2，它基于 UDP/QUIC，在我的网络环境中表现明显更好。

这里不展开具体节点凭据，只说明当时解决问题的顺序：

1. 安装 Hysteria2 服务端。
2. 为域名申请 TLS 证书。
3. 配置认证和混淆。
4. 将服务放到高位 UDP 端口。
5. 增大系统的 UDP 接收和发送缓冲区。
6. 在客户端分别测试延迟、下载速度和实际视频播放。

过程中遇到的第一个错误是：服务明明安装好了，却无法启动。

查看日志：

```bash
systemctl status hysteria-server
journalctl -u hysteria-server --no-pager -n 50
```

日志提示证书文件没有读取权限。原因是证书原本放在 root 的私人目录，而 Hysteria2 使用普通系统用户运行。

我把证书复制到服务自己的目录，并调整所有者：

```bash
mkdir -p /etc/hysteria/cert
cp /你的证书目录/fullchain.pem /etc/hysteria/cert/
cp /你的证书目录/privkey.pem /etc/hysteria/cert/
chown -R hysteria:hysteria /etc/hysteria/cert
chmod 600 /etc/hysteria/cert/privkey.pem
```

重启后服务才正常运行：

```bash
systemctl restart hysteria-server
systemctl is-active hysteria-server
```

经过几轮调整，实际速度从最初的 20kbps 提高到了可以正常观看高清视频的水平。

这段经历教会我一件很重要的事：**程序启动失败时先读日志，不要不断重装。** 日志里通常已经写出了真正原因。

## 六、第二个问题：服务正常，为什么还是连不上

运行一段时间后，连接突然变得不稳定。

奇怪的是，服务器还能被 ping 通，面板里也显示服务进程正在运行，但 SSH 和实际数据连接经常失败。

刚开始我一直重启服务，后来才发现判断方式错了。

### ping 通不代表服务能访问

`ping` 使用的是 ICMP，而 SSH、网页和大多数应用使用 TCP 或 UDP。即使 ping 有响应，也不能说明某个 TCP 或 UDP 端口正常。

排查时应该分层检查：

```text
1. VPS 是否开机
2. 服务进程是否运行
3. 服务是否监听正确端口
4. 防火墙是否放行
5. 从外部能否连接该端口
6. DNS 和 CDN 配置是否正确
```

检查服务是否运行：

```bash
systemctl is-active 你的服务名
```

检查端口是否真的在监听：

```bash
ss -tlnp
ss -ulnp
```

检查防火墙：

```bash
ufw status numbered
```

这次最终确认并不是应用程序崩溃，而是直连网络出现异常。也就是说，继续重装服务不会有用。

## 七、为什么还要准备备用线路

只准备一个连接方式时，一旦直连线路出问题，我连管理和排查都会很困难。

后来我增加了一条经过 Cloudflare 的备用线路。它不一定比直连更快，但在直连异常时可以提供另一个入口。

目前我的结构是：

```text
日常使用：VLESS + WebSocket + TLS → 直接连接 VPS
异常备用：VLESS + WebSocket + TLS → Cloudflare → VPS
Hysteria2：配置保留，但目前暂停运行
```

为什么暂停曾经速度很快的 Hysteria2？因为我后来更看重稳定性和客户端兼容性。最快的方案不一定最适合长期使用，尤其当它需要额外维护端口范围、UDP 规则和不同客户端配置时。

### 我配置 Cloudflare 时踩的坑

Cloudflare 不是打开橙色云就一定能用。至少需要同时检查：

- 使用的端口是否属于 Cloudflare 支持的代理端口；
- 客户端连接的域名是否开启代理；
- WebSocket 的 Host 是否正确；
- TLS 的 SNI 是否与证书匹配；
- Cloudflare 的 SSL 模式是否与服务器回源方式一致。

我曾经为了排查问题连续改了端口和 TLS 设置，结果旧问题没有解决，反而把原本可用的备用线路改坏了。最后只能回到服务端逐项确认实际配置和监听端口。

现在再遇到类似问题，我会一次只改一个变量，修改后立即验证，而不是同时改好几项。

## 八、安装完成后，安全才刚开始

最初我觉得只要密码足够复杂就安全。后来查看登录日志，才发现公网服务器会不断收到自动扫描和登录尝试。

### 1. 防火墙默认拒绝入站

我使用 UFW，只开放正在使用的端口：

```bash
ufw default deny incoming
ufw default allow outgoing

# 示例：先放行新的 SSH 端口
ufw allow 你的SSH端口/tcp

ufw enable
ufw status numbered
```

这里不要照抄某个教程的全部端口。每多开放一个端口，就多一个需要维护的公网入口。

### 2. 把 SSH 改为密钥登录

密码可能被猜测，SSH 私钥则是一份只保存在自己设备上的身份凭证。

在 Windows 上生成 Ed25519 密钥：

```powershell
ssh-keygen -t ed25519
```

把公钥添加到服务器后，先开一个新的终端窗口测试密钥是否能成功登录。确认没问题，再关闭 SSH 密码认证。

修改 SSH 配置前，我现在一定会保留原连接，并准备好服务商的网页控制台。因为一旦配置写错，控制台可能是唯一的修复入口。

### 3. 使用 fail2ban

fail2ban 会读取登录失败日志。当同一个地址在短时间内多次失败，它会临时封禁这个来源。

```bash
apt install fail2ban -y
systemctl enable --now fail2ban
fail2ban-client status sshd
```

它不能代替密钥和防火墙，但可以减少持续爆破带来的风险和日志噪声。

### 4. 管理面板不再直接暴露公网

我最初为了方便，直接在公网开放管理面板。后来将它改为只允许 VPS 本机访问，平时通过 SSH 隧道连接：

```powershell
ssh -L 本地端口:127.0.0.1:面板端口 root@服务器地址 -p SSH端口
```

保持这个 SSH 窗口开启，然后在浏览器访问 `https://127.0.0.1:本地端口/`。

这样，攻击者在公网连管理面板的登录页面都看不到。对我来说，只是使用前多打开一个隧道窗口，安全收益却很明显。

## 九、为什么需要自动备份

经历过一次配置故障后，我开始认真做备份。

最早的备份方式是：想起来时手动复制一次。这种方式最大的问题就是，真正出事时手里的备份往往已经很旧。

现在服务器会定时打包这些关键内容：

- 服务数据库和配置文件；
- TLS 证书；
- SSH 和防火墙的重要配置；
- 自动化脚本及计划任务。

本地只保留最近一段时间的版本，避免备份无限增长占满硬盘。另外还有一份加密后的副本保存在自己的 Windows 电脑上。

### 有文件不等于备份可用

我还增加了自动检查，确认：

1. 最近一次备份没有过期；
2. 文件大小没有异常；
3. 压缩包能够读取；
4. 关键文件确实在压缩包里。

我也实际测试过异地备份能否解密。否则等到服务器坏了才发现备份打不开，就已经来不及了。

## 十、让服务器主动告诉我它出问题了

以前我只有在服务无法使用时才知道服务器出了问题。现在 VPS 会定时检查自己的状态，并通过消息机器人发送通知。

它会检查：

- 服务进程是否运行；
- 使用中的端口是否监听；
- CPU、内存和磁盘是否异常；
- TLS 证书是否快过期；
- 最近的备份是否正常；
- SSH 是否出现大量失败登录；
- 系统是否有等待安装的更新。

对于明确的进程停止，脚本会尝试重启；但端口异常或资源过高时只告警，不会随便改配置。

这是我后来才想明白的：**自动化应该处理确定的事情，而不是自作聪明地修复所有问题。**

### 服务器自己报警也有盲区

如果只是某个服务挂了，VPS 还可以发送消息。但如果整台 VPS 断电或者完全断网，它连告警都发不出来。

因此，服务器内部监控只能解决一部分问题。后续我还准备增加一个独立于 VPS 的外部监控，从另一台机器定时检查网站或端口是否可访问。

## 十一、日志也会占满硬盘

监控和自动任务运行久了会不断写日志。如果完全不管，小硬盘 VPS 可能被日志占满。

我为系统日志和自定义脚本设置了轮转规则：

- 单个日志达到一定大小后自动切分；
- 旧日志压缩保存；
- 只保留有限数量；
- 系统日志设置总大小和最长保存时间。

这是一件平时感觉不到、但很适合提前做的维护工作。

## 十二、现在我怎么排查故障

以前出问题时，我会到处复制命令。现在会按固定顺序检查。

### 第一步：服务器是否在线

先看服务商后台的电源状态。如果 SSH 完全连不上，尝试网页控制台。

### 第二步：服务是否运行

```bash
systemctl is-active 你的服务名
systemctl status 你的服务名
```

### 第三步：端口是否监听

```bash
ss -tlnp
ss -ulnp
```

进程显示运行，却没有监听预期端口，通常说明配置没有正确加载。

### 第四步：查看日志

```bash
journalctl -u 你的服务名 --no-pager -n 100
```

重点看第一次出现的错误，不要只看最后一行“启动失败”。

### 第五步：检查防火墙和外部网络

```bash
ufw status numbered
```

如果服务、端口和防火墙都正常，但外部依然无法连接，再考虑 DNS、CDN 或线路问题。

这套顺序最大的好处是：不会在网络出问题时反复重装程序，也不会在程序配置错误时误以为服务器 IP 出了问题。

## 十三、这次经历里最重要的几个坑

| 问题 | 当时的错误判断 | 后来的解决办法 |
|---|---|---|
| 速度只有 20kbps | 便宜 VPS 性能差 | 先测服务器带宽，再排查协议和网络路径 |
| 服务启动失败 | 安装过程出错 | 查看日志，修复证书文件权限 |
| ping 通但连不上 | 服务肯定坏了 | 分别检查 TCP、UDP、端口和防火墙 |
| SSH 改端口后断连 | SSH 服务挂了 | 先放行新端口、测试，再关闭旧入口 |
| CDN 备用线路失效 | Cloudflare 不稳定 | 逐项核对端口、Host、SNI、TLS 和回源 |
| 配置改了但没生效 | 数据库没保存 | 检查服务是否重新加载及真实监听状态 |
| 有备份文件 | 已经万无一失 | 定期验证压缩包内容并测试解密恢复 |

## 最后的感受

如果只是复制命令，一台 VPS 可能半小时就能“搭好”。但要让它安全、稳定，并且出问题后知道怎样修复，花的时间会更多。

我现在仍然算不上专业运维，只是比刚开始多了一些经验：

- 修改前先确认现状；
- 一次只改一件事；
- 修改后检查进程、端口和日志；
- 不把管理入口全部暴露在公网；
- 备份必须经过恢复验证；
- 最快的方案不一定最适合长期使用。

这台 VPS 现在已经从一个“能运行服务的小盒子”，变成了我学习 Linux、网络和安全的实验场。以后遇到新的问题，我也会继续把过程补充在这里。
